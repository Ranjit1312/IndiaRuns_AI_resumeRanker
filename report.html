Current Architecture

The harness implements a Depth-1 Recursive Language Model (RLM) pattern: deterministic code acts as the root, dispatching focused leaf LLM calls per field, then validating and repairing the assembled profile. resume.py mirrors coerce.py for the résumé path, sharing the same llm_query / Environment / HarnessLogger spine.

Friction Inventory — 6 Deepening Opportunities

#1 ▶ Parallel _BUILDERS + _SENTINELS dicts in coerce.py HIGH

Today — Shallow

Two separate module-level dicts, keyed by identical string literals. Adding a new builder requires editing three places: the function, _BUILDERS, and _SENTINELS. They can silently diverge (a builder with no sentinel, or vice-versa). The repair loop then contains a fragile if top in _BUILDERS … if top in _SENTINELS cascade.

if top in _BUILDERS
if top in _SENTINELS"] B --> R S --> R

After — Deeper

A single @dataclass BlockSpec per field: name, builder_fn, sentinel_fn. One BLOCK_REGISTRY: list[BlockSpec] replaces both dicts. The repair loop becomes a simple spec.builder(…); spec.sentinel(…). Adding a new field is one dataclass instantiation. Divergence is impossible.

# NEW interface (harness/coerce.py)
@dataclass
class BlockSpec:
    name: str
    builder: Callable[..., Any]
    sentinel: Callable[[dict], Any]
BLOCK_REGISTRY: list[BlockSpec] = [
    BlockSpec("role",                 build_role,           _sentinel_role),
    BlockSpec("locations",            build_locations,      lambda _: dict(F.DEFAULT_LOCATIONS)),
    BlockSpec("signals",              build_signals,        _sentinel_signals),
    BlockSpec("domain",               build_domain,         lambda _: dict(F.DEFAULT_DOMAIN)),
    BlockSpec("relevant_skill_regex", build_relevant_skill, lambda _: F.DEFAULT_RELEVANT_SKILL_REGEX),
    BlockSpec("red_flags",            build_red_flags,      lambda _: F.sanitize_red_flags(None)),
]
_SPEC_MAP = {s.name: s for s in BLOCK_REGISTRY}   # O(1) lookup, same as before

Locality gain: every future field is one dataclass line; the repair loop never needs editing again.

#2 ▶ validate_profile_dict round-trips through a temp file on every call HIGH

Today — Shallow

validate_profile_dict in harness/validate.py serialises the dict to YAML, writes it to tempfile.NamedTemporaryFile, then calls rprofile.load(tmp.name) which reads the file back. This is called up to 6 times per compile run (initial + up to 2 repairs × 2 sentinels each). On read-only filesystems (HF Spaces containers) this silently fails. It is also the main reason adding a new field breaks if jd/method_config.yaml path is wrong — the error is a file-not-found that bubbles as a sentinel, not a clear message.

After — Deeper

Expose a rprofile.load_dict(d, method_path) seam in redrob_ranker/profile.py that accepts a Python dict directly. If the engine cannot be modified, add a _validate_in_process(prof) helper in harness/validate.py that calls yaml.safe_dump into a io.StringIO then yaml.safe_load back — removing the filesystem roundtrip entirely. Filesystem is only needed for the full engine path.

# harness/validate.py (after)
import io
def validate_profile_dict(prof, method_path=METHOD_PATH):
    buf = io.StringIO()
    yaml.safe_dump(prof, buf, sort_keys=False, allow_unicode=True)
    buf.seek(0)
    # rprofile.load_dict() — new seam in the engine, or:
    import tempfile, os
    # fallback: still write tmp, but to the OS temp dir, not cwd
    with tempfile.NamedTemporaryFile("w", suffix=".yaml",
            dir=tempfile.gettempdir(), delete=False, encoding="utf-8") as f:
        f.write(buf.getvalue())
        fname = f.name
    try:
        rprofile.load(fname, method_path)
        return ValidationResult(ok=True)
    …

Locality gain: validation logic is in one place; HF Spaces read-only FS stops being a latent failure mode.

#3 ▶ health is an untyped mutable dict threaded through 6+ functions MEDIUM

Today — Shallow

The health dict is initialised in compile_jd() as {"defaulted": [], "sentineled": [], "repairs": 0, …} and passed by reference into every build_* function which mutates it. There is no type annotation on what keys are expected — nothing prevents a typo (health["deafulted"]) from silently creating a new key instead of appending to the correct list.

After — Deeper

Replace with a @dataclass CompileHealth (or a TypedDict for zero-migration cost). Builders receive it and call health.defaulted.append("role") — a typo now raises AttributeError immediately in tests. The dataclass also exports .as_dict() for the JSON health tab.

@dataclass
class CompileHealth:
    defaulted:  list[str] = field(default_factory=list)
    sentineled: list[str] = field(default_factory=list)
    repairs:    int = 0
    model:      str = ""
    metadata:   dict = field(default_factory=dict)
    telemetry:  dict | None = None
    def as_dict(self) -> dict:
        return dataclasses.asdict(self)

Locality gain: every builder's interface is narrower (typed health param); tests can instantiate CompileHealth() directly without constructing a magic dict.

#4 ▶ coerce.py and resume.py duplicate the repair-loop pattern verbatim MEDIUM

Today — Shallow

Both files implement the same validate → if failing top-block → re-call builder → apply sentinel → loop pattern independently. Any bug in the repair logic (e.g., the current bare except Exception: pass on line 221 of coerce.py) must be fixed in two places. The docstrings even say "mirrors coerce.py's discipline."

After — Deeper

Extract a repair_loop(profile, registry, env, backend, health, validator, …) function into harness/rlm.py (the engine spine). Both compile_jd and compile_resume call it with their own registries and validators. The loop is written once, tested once, fixed once.

Locality gain: repair-loop tests cover both JD and résumé paths; the bare except: pass is fixed in one place.

#5 ▶ Environment lives in rlm.py but is a pure data/parsing class MEDIUM

Today — Shallow

rlm.py mixes two very different concerns: (a) JD section parsing / snippet slicing (Environment) and (b) the LLM network transport + retry (llm_query). Both coerce.py and resume.py import from rlm, coupling the JD parsing logic to the transport layer. Tests that only want to exercise snippet slicing must import the transport layer too.

After — Deeper

Move Environment (and _SECTION_KEYS, _HEADER_RE, MAX_SNIPPET) to a new harness/document.py. rlm.py becomes a pure transport module (llm_query only). The import graph becomes a DAG with no mixed concerns. test_document.py can verify section detection without importing any backend.

Locality gain: section-detection bugs are isolated to document.py; transport bugs to rlm.py. Both are testable independently with zero mocking.

#6 ▶ app.py is 1,300 lines — UI, business logic, and eval orchestration in one file MEDIUM

Today — Shallow

app.py has grown to 1,317 lines containing: JD compilation UI, résumé scoring UI, HITL form, eval study runner, store bootstrap, and utility helpers — all in one flat script. The Streamlit rerun model means every helper is re-executed on every interaction. Testing any single flow requires simulating the full Streamlit session state.

After — Deeper

Split into page modules: pages/jd_compiler.py, pages/resume_scorer.py, pages/eval_study.py. Keep app.py as a thin entrypoint (sidebar, session bootstrap, page routing). Business logic (store queries, compile calls) moves to ui_helpers.py. Each page module is independently testable with a MockStore.
